---

## 1. PRD란 무엇인가

PRD는 **Product Requirements Document**의 약자이며, 제품이 **왜 필요한지**, **무엇을 해야 하는지**, **어떤 사용자 문제를 해결해야 하는지**, **어떤 범위까지 제공해야 하는지**를 정의하는 문서이다.

PRD는 코드 설계서가 아니며, 구현 세부 기술을 확정하기 전에 다음을 먼저 고정하기 위한 문서이다.

- 제품 목표
- 사용자 문제
- 핵심 기능 범위
- 사용 시나리오
- 제약 조건
- 성공 기준

즉, PRD는 “어떻게 코딩할 것인가”보다 먼저 “무엇을 제품으로 만들 것인가”를 결정하는 기준 문서이다.

---

## 2. 기능명

**AI 기반 실행형 절차 생성 및 진행 기능**  
영문명: **AI-driven Executable Workflow Mindmap**

---

## 3. 기능 한 줄 정의

사용자가 자연어로 요청한 작업을 AI가 **step 기반 mindmap**으로 구조화하고, 사용자는 각 step를 실제로 실행하면서 오류를 해결하며, 최종적으로는 시행착오를 정리한 **최종 실행 절차서 형태의 map**을 완성할 수 있어야 한다.

---

## 4. 배경과 문제 정의

기존의 일반 대화형 AI 사용 방식에는 다음 문제가 있다.

1. 전체 절차가 긴 채팅 형태로 제공되어 흐름을 잃기 쉽다.
2. 각 step에서 오류가 발생하면 질의응답이 길어져 원래 절차 구조가 무너진다.
3. 오류 해결 과정이 누적되면 전체 작업의 현재 위치를 파악하기 어렵다.
4. 최종적으로는 성공한 방법보다 실패했던 대화 로그가 더 많이 남는다.
5. 반복 가능한 표준 절차서로 재사용하기 어렵다.

즉, 기존 방식은 “답변은 길어지는데 절차 구조는 유지되지 않는 문제”가 있다.

---

## 5. 제품 목표

본 기능의 목표는 다음과 같다.

1. 긴 기술 절차를 **step 구조**로 분해한다.
2. 각 step를 **node 단위**로 관리한다.
3. 긴 설명과 명령어는 **note**에 저장한다.
4. 사용자는 현재 step를 실제 실행하고 결과를 바탕으로 다음 step로 넘어간다.
5. 오류가 발생하면 같은 step 문맥에서 해결한다.
6. 해결이 끝나면 실패 이력은 정리하고 **최종 성공 절차만 남긴다**.
7. 결과적으로 map이 “아이디어 맵”이 아니라 **실행 가능한 절차 문서**가 되도록 한다.

---

## 6. 핵심 컨셉

이 기능은 일반적인 “AI가 mindmap을 만들어 주는 기능”과 다르다.

### 일반 AI mindmap 생성
- 주제를 요약해 구조화
- 정적인 결과물 생성
- 보기 중심

### 본 기능의 AI
- 작업 절차를 단계별로 구조화
- 실제 실행을 전제로 함
- 오류 해결을 같은 step 안에서 반복 가능
- 최종적으로 정제된 절차를 유지함
- 실행 중심 / 운영 중심 / 개발 중심

따라서 본 기능은 다음으로 정의할 수 있다.

> **AI 대화 결과를 map으로 보여주는 기능이 아니라, map을 중심으로 실제 작업을 진행하고 정제해 나가는 실행형 AI 절차 기능**

## 8. 대표 사용 시나리오

### 시나리오 A: 서버 설치 절차 생성
사용자가 “Ubuntu 22.04에 Apache 설치 및 Let’s Encrypt SSL 발급 절차를 초보자 수준으로 설명해줘”라고 요청한다.

AI는 다음을 수행한다.
- 전체 절차를 step별 node로 생성
- 각 node 제목은 짧게 작성
- 명령어/상세 설명/주의사항은 note에 저장

사용자는 Step 1부터 실제 실행한다.

### 시나리오 B: step 실행 중 오류 해결
사용자가 특정 step의 명령어 실행 후 오류 메시지를 붙여넣는다.

AI는 다음을 수행한다.
- 해당 step 문맥에서 원인 분석
- 해결 명령어 또는 수정 절차 제안
- 필요 시 임시 하위 node 또는 보조 note 생성

### 시나리오 C: 최종 정제
오류 해결 과정에서 여러 임시 시도가 있었더라도, 최종적으로는 다음 상태가 되어야 한다.
- 원래 step node는 최종 성공 버전으로 업데이트
- 실패한 중간 시도 node는 삭제 또는 접기/숨김 처리
- map에는 최종 실행 절차만 남음

## 9. 기능 범위

### 9.1 포함 범위
1. AI가 자연어 요청을 절차형 map으로 생성
2. step 기반 node 생성
3. 긴 설명/명령어를 note로 저장
4. step별 실행 결과 기반 진행
5. 오류 발생 시 동일 step 문맥에서 AI 질의응답
6. 최종 해결안 중심으로 map 정제
7. step 상태 관리
8. note 내 code block 지원
9. code block copy 버튼 지원
10. 단독 편집 모드에서만 AI 기능 허용

### 9.2 제외 범위 (현재 PRD 기준)
1. 협업 중 AI 자동 생성/확장 허용
2. AI와 협업 엔진의 동시 충돌 자동 병합
3. 실패 이력 전체를 영구 보존하는 감사 로그 기능
4. 코드 실행 자체를 제품 내부에서 수행하는 기능
5. 실시간 원격 터미널 기능

## 10. 상세 기능 정의

### 10.1 AI 절차형 map 생성
사용자의 자연어 요청을 입력으로 받아 전체 작업을 step 기반 구조로 생성한다.

#### 생성 원칙
- 하나의 step는 하나의 node
- node 제목은 짧고 식별 가능해야 함
- 긴 설명은 note로 분리
- step 순서가 명확해야 함
- 필요 시 서브 step 허용

#### 출력 예시
- Step 1: 패키지 업데이트
- Step 2: Apache 설치
- Step 3: 서비스 확인
- Step 4: 방화벽 설정
- Step 5: Certbot 설치
- Step 6: SSL 발급

---

### 10.2 Note 중심 상세 정보 저장
각 step node는 상세 내용을 담는 note를 가진다.

#### note에 포함 가능한 정보
- 설명 텍스트
- 명령어
- 설정 예시
- 주의사항
- 예상 결과
- 확인 방법
- 오류 메시지 예시
- 해결 방안

#### note 원칙
- node는 짧고 구조적이어야 한다.
- note는 길고 실무적인 내용을 담는다.
- node와 note가 역할을 분리한다.

---

### 10.3 Note 내 code block 지원
note는 단순 문자열이 아니라 구조화된 rich note를 지원해야 한다.

#### 최소 지원 블록 타입
- paragraph
- bullet list
- code block
- warning / tip / callout
- checklist (추후 확장 가능)

#### code block 요구사항
- 언어 타입 지정 가능
  - bash
  - shell
  - sql
  - json
  - yaml
  - javascript
  - typescript
  - python
  - plaintext
- code block별 Copy 버튼 제공
- note 안에 여러 code block 삽입 가능
- 사용자는 개별 code block만 복사 가능해야 한다.

#### 목적
- 명령어 실행 정확도 향상
- 초보자의 복사/붙여넣기 실수 감소
- 실행형 절차 문서의 사용성 강화

---

### 10.4 step 단위 진행
사용자는 전체 map을 한 번에 읽는 것이 아니라 현재 step를 기준으로 작업을 수행한다.

#### 진행 방식
- 현재 step 선택
- note의 명령어 확인 및 복사
- 실제 환경에서 실행
- 결과 확인
- 성공 시 다음 step로 이동

즉, map은 정적 문서가 아니라 “따라가는 절차”가 된다.

---

### 10.5 오류 해결형 step 운영
실행 중 오류가 발생하면, 사용자는 같은 step 문맥에서 오류 내용을 입력하고 AI에게 해결을 요청할 수 있어야 한다.

#### 동작 방식
- 오류 메시지 캡처/텍스트 입력
- AI가 현재 step의 원래 목적과 문맥을 유지한 채 분석
- 수정 명령어 또는 대체 절차 제안
- 필요 시 보조 설명 추가

#### 핵심 원칙
오류 해결은 전체 대화의 새로운 주제가 아니라, **현재 step의 일부 작업**으로 처리되어야 한다.

---

### 10.6 절차 정제 기능
본 기능의 가장 중요한 차별 요소 중 하나는 **중간 실패 이력을 그대로 누적하지 않고 최종 성공 절차 중심으로 재정리하는 것**이다.

#### 예시
Step 3에서 원래 명령이 실패했고, 추가 질문과 수정 명령어가 3번 있었더라도:
- 실패했던 중간 하위 node들은 정리 대상
- 최종적으로 성공한 해결안만 Step 3 note 또는 최종 하위 구조에 반영

#### 정리 원칙
- map은 항상 “최종 실행 가능한 절차”를 우선한다.
- 장황한 실패 이력은 기본 뷰에 남기지 않는다.
- 필요하면 히스토리 계층으로 숨길 수 있으나, 기본 구조는 정제된 상태를 유지한다.

---

### 10.7 step 상태 관리
각 step는 상태를 가져야 한다.

#### 기본 상태
- not-started
- in-progress
- blocked
- resolved
- done

#### 목적
- 현재 어디까지 진행했는지 파악
- 어느 step에서 막혔는지 표시
- 작업 재개 시 현재 위치를 빠르게 찾기 위함

---

### 10.8 AI 기능 사용 정책
AI 기능은 동시 협업 충돌을 줄이기 위해 **단독 편집 상태에서만** 허용한다.

#### 정책
- 맵을 연 사용자가 1명일 때: AI 기능 사용 가능
- 동일 맵을 2명 이상이 동시에 열고 있을 때: AI 기능 사용 제한

#### 제한 대상
- AI Generate
- AI Expand
- map 구조를 변경하는 AI 반영 기능

#### UI 원칙
- 협업 상태에서는 AI 버튼 비활성화 또는 숨김
- 안내 문구 표시
  - 예: “협업 중에는 AI 자동 생성/확장 기능을 사용할 수 없습니다.”

## 11. 사용자 가치

### 11.1 개인 사용자 가치
- 긴 기술 절차를 한눈에 볼 수 있다.
- 현재 step만 집중해서 실행할 수 있다.
- 오류 해결 과정이 구조를 무너뜨리지 않는다.
- 최종 절차가 깔끔하게 남는다.

### 11.2 조직 사용자 가치
- 실행 가능한 표준 절차서 축적
- 온보딩 문서 품질 향상
- 운영 runbook 고도화
- 팀 내 절차 재사용성 증가
- 기술 문서의 최신성 유지

---

## 12. 대규모 애플리케이션 개발에 대한 가치

이 기능은 서버 설치나 단일 운영 작업뿐 아니라 대규모 애플리케이션 개발에서도 유용하다.

### 적용 가능 영역
- 기능 개발 절차
- DB 변경 절차
- 릴리즈 체크리스트
- API 추가 절차
- 테스트 절차
- 장애 대응 절차
- 운영 점검 절차

### 이유
1. 긴 개발 절차를 구조화할 수 있다.
2. step별 명령어/쿼리/설정 예시를 note로 정리할 수 있다.
3. 오류 해결을 같은 step 안에서 정리할 수 있다.
4. 최종적으로 팀이 재사용 가능한 절차서가 남는다.

즉, 이 기능은 단순한 아이디어 정리 도구를 넘어 **개발/운영 workflow 관리 도구**로 확장 가능하다.

## 13. 제품 포지셔닝

기존의 easymindmap가 단순한 mindmap 편집기였다면, 본 기능이 추가되면 제품의 성격은 다음처럼 확장된다.

### 기존 포지셔닝
- 아이디어 정리 도구
- 시각적 구조화 도구

### 확장 포지셔닝
- AI 기반 실행형 절차 관리 도구
- 실행 가능한 기술 문서 생성 도구
- 개발/운영 runbook builder
- 대화형 workflow mindmap 도구

---

## 14. 성공 기준

다음 항목을 만족하면 본 기능이 유의미하게 구현된 것으로 본다.

1. 사용자가 자연어 요청만으로 절차형 map을 생성할 수 있다.
2. 각 step에 note와 code block을 둘 수 있다.
3. code block은 개별 copy가 가능하다.
4. 사용자는 step별로 실행하고 다음 단계로 진행할 수 있다.
5. 오류 해결이 같은 step 문맥에서 반복 가능하다.
6. 해결 후 map이 최종 절차 중심으로 정리된다.
7. 협업 중에는 AI 기능이 제한된다.

---

## 15. 핵심 차별점 요약

본 기능의 핵심 차별점은 다음과 같다.

1. AI가 “답변”이 아니라 “절차 구조”를 만든다.
2. 긴 내용은 note로 분리하여 구조 가독성을 유지한다.
3. code block copy 지원으로 실행성을 높인다.
4. 오류 해결이 step 문맥 안에 흡수된다.
5. 최종적으로는 실패 로그가 아니라 성공 절차가 남는다.

---

## 16. 최종 정의

> easymindmap의 AI 기반 실행형 절차 Mindmap 기능은, 사용자의 자연어 요청을 step 구조의 실행 가능한 map으로 만들고, 각 step의 note에 명령어와 상세 설명을 담으며, 실행 중 발생한 오류를 같은 step 문맥 안에서 해결하고, 최종적으로 정제된 절차 문서로 완성하는 기능이다.
