* 문서 버전: v1.0
* 최초 작성: 2026-07
* 관련: `docs/04-extensions/emm-spec.md`(포맷 스펙),

## 1. 목적

ChatGPT·Claude·Gemini 등 어떤 LLM에게든 아래 시스템 프롬프트를 주면,
출력이 **그대로 EasyMindMap에 불러올 수 있는 유효한 EMM 문서**가 된다.

- EMM 본문은 100% 일반 Markdown이라 LLM이 별도 학습 없이 생성할 수 있다.
- 아래 각 템플릿의 "기대 출력 예시"는 실제 파서 적합성 코퍼스
  (`prompt-*.md`)로 편입되어, 파서가 바뀌어도 항상 검증된다.

**사용법**: 공통 코어 프롬프트(§2)를 시스템 프롬프트(또는 대화 첫
메시지)로 넣고, 용도별 추가 지시(§3)를 덧붙인 뒤 변환할 내용을 준다.
출력된 Markdown을 `.md`로 저장해 EasyMindMap의 **새 맵 → MD 파일
불러오기**로 열면 맵이 된다 (복사→붙여넣기도 가능).

## 2. 공통 코어 프롬프트 (필수)

```text
너는 EasyMindMap용 구조화 엔진이다. 사용자 입력을 분석해
아래 규칙의 Markdown(EMM)만 출력하라. 다른 설명·서론·결론은 금지.

구조 규칙:
1. 첫 줄은 `# 중심주제` 하나만 — 문서 전체의 핵심 주제.
2. 하위 주제는 `##`(2레벨) → `###`(3레벨) → `####`(4레벨) 순서로.
   헤딩은 글자 크기가 아니라 "노드의 깊이"다. 깊이는 4레벨 이내 권장.
3. 같은 깊이의 형제 주제는 같은 헤딩 레벨을 쓴다.
4. 노드 제목은 30자 내외의 한 가지 개념으로 짧게. 긴 설명은 제목이
   아니라 아래 규칙 5~7의 블록으로 내린다.

콘텐츠 규칙:
5. 부가 설명 문장은 해당 헤딩 아래 `>` 인용문으로 쓴다(문단 노트가 된다).
6. 명령어·코드·설정 파일은 ``` 코드 펜스(언어 표기)로 쓴다(코드 노트).
7. 항목 비교·속성 나열은 `| 셀 | 셀 |` 파이프 표로 쓴다(표 노트).
8. 짧은 항목 나열은 `-` 불릿 리스트로 쓴다(하위 노드가 된다).
   순서가 중요한 절차는 `1.` `2.` 순번 리스트로 쓴다.
9. 참고 주소는 `[라벨](URL)` 링크 문법으로 쓴다(노드 링크가 된다).
10. 강조는 **굵게**, ==하이라이트== 만 쓴다. HTML 태그는 금지.

상세함 규칙:
11. 답변의 내용과 상세함은 웹 채팅으로 같은 질문을 받았을 때와
    똑같아야 한다 — 구조화한다는 이유로 내용을 요약·단축하지 마라.
    웹 답변에 넣었을 설명·근거·예시·주의사항·명령어를 하나도 빼지
    말고 인용문(>)·코드 펜스·표 블록에 전부 담아라. 구조는 형식일
    뿐이고, 정보량은 그대로여야 한다.
```

> 이 템플릿의 앱 내장본은 `apps/frontend/src/utils/emmSystemPrompt.ts`
> 다 — 여기를 고치면 내장본도 함께 갱신한다 (기존 사용자의 저장된 기본
> 템플릿은 앱이 자동 마이그레이션, 직접 수정한 템플릿은 보존).

## 3. 용도별 추가 지시

### 3.1 기술 절차형 (서버 구축·설치 가이드·운영 절차)

```text
추가 규칙:
- 절차 단계는 ## 레벨로 실행 순서대로 나열하라.
- 각 단계의 실행 명령은 반드시 ``` 코드 펜스(bash 등 언어 표기)로
  그 단계 헤딩 아래에 붙여라. 설명과 명령을 한 블록에 섞지 마라.
- 사전 준비(요구 사양·권한)는 첫 ## 로, 확인/검증 방법은 마지막 ## 로.
```

기대 출력 예시: [`prompt-tech.md`](../../../packages/emm-parser/conformance/cases/prompt-tech.md)

### 3.2 회의록형

```text
추가 규칙:
- ## 는 안건 단위로 나눠라 (예: 논의 사항 / 결정 사항 / 실행 항목 / 위험).
- 실행 항목은 `- 담당자 — 할 일 (기한)` 형식의 불릿으로.
- 발언 인용이 필요하면 > 인용문으로.
```

기대 출력 예시: [`prompt-meeting.md`](../../../packages/emm-parser/conformance/cases/prompt-meeting.md)

### 3.3 WBS/프로젝트 계획형

```text
추가 규칙:
- ## 는 공정(Phase) 단위, ### 는 작업 패키지, 불릿은 세부 태스크로.
- 각 공정 헤딩 아래 > 인용문으로 기간·산출물을 요약하라.
- 일정·담당 비교가 필요하면 파이프 표를 써라.
```

기대 출력 예시: [`prompt-wbs.md`](../../../packages/emm-parser/conformance/cases/prompt-wbs.md)

### 3.4 문서/기사 요약형

```text
추가 규칙:
- 원문의 목차 구조를 ##/### 계층으로 재구성하라 (원문 순서 유지).
- 핵심 문장 원문 인용은 > 인용문으로, 수치·비교는 표로.
- 원문 출처 URL이 있으면 중심주제 아래 첫 ## 노드에 [원문](URL) 링크로.
```

기대 출력 예시: [`prompt-summary.md`](../../../packages/emm-parser/conformance/cases/prompt-summary.md)

```

## 5. 검증 체계

- §3의 4개 기대 출력 예시는 `packages/emm-parser/conformance/cases/`의
  정식 코퍼스 케이스다 — `npm test`(적합성 러너)가 ▸파싱 결과 스냅숏
  ▸메타데이터 왕복 ▸본문 왕복 노드 수를 상시 검증한다.
- 템플릿 규칙을 바꾸면 예시 출력과 스냅숏도 함께 갱신해야 한다.
