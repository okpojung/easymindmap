* 명칭: **EMM — EasyMindMap Markdown**
* 스펙 버전: **v1.0 (Draft)**

* 최초 작성: 2026-07

## 0. 이 문서의 위상

EasyMindMap은 단순한 마인드맵 작성 도구가 아니라, **마인드맵을 위한
Markdown 포맷(EMM)을 정의·표준화하고 이를 편집·렌더링하는 레퍼런스
구현**을 지향한다. 이 문서는 그 포맷의 정식 명세다.

- `markdown-export.md`는 이 스펙의 **구현 규칙 상세**(변환 표 단위)이며,
  구현이 바뀔 때마다 갱신되는 살아있는 문서다.
- 본 스펙은 포맷의 **설계 원칙·문법·적합성 기준·버전 정책**을 정의하는
  상위 문서다. 두 문서가 충돌하면 본 스펙 개정을 통해 해소한다.

## 1. 배경과 목표

### 1.1 시장 공백

마인드맵을 표현하는 텍스트 포맷의 사실상 표준은 아직 없다.

| 항목 | Mermaid (mindmap) | Markmap | FreeMind/Freeplane | **EMM** |
|---|---|---|---|---|
| 기반 | MD 코드블록 | MD 헤딩 | XML | **CommonMark/GFM 본문 + 메타데이터 계층** |
| 구조 | 트리 | 트리 | 트리 | 트리 (+확장 예정: 크로스 엣지) |
| 노드 콘텐츠(노트·표·코드) | 제한 | 없음 | 있음 | ✅ 문단·표·코드·체크리스트 노트 |
| 레이아웃 | ❌ | ❌ | 제한 | ✅ 방사형·트리·계층·진행트리·타임라인 + 레벨/서브트리 오버라이드 |
| 스타일(도형·색·폰트) | 제한 | ❌ | 있음 | ✅ 레벨 정책 + 노드 오버라이드 |
| 사진(위치 보존) | ❌ | ❌ | 제한 | ✅ 내장(base64) + 텍스트 중간 위치 |
| 일반 MD 뷰어에서 가독 | 코드블록으로 보임 | ✅ | ❌ | ✅ **본문이 순수 GFM** |
| 무손실 왕복(라운드트립) | ❌ | ❌ | 자체 포맷 내 | ✅ 메타데이터 계층으로 보장 |
| AI 생성 친화성 | 중간 | 높음 | 낮음 | 높음 (본문 = 일반 MD) |

### 1.2 목표

1. **사람이 읽을 수 있어야 한다** — EMM 파일은 어떤 Markdown
   뷰어(GitHub·Obsidian·Notion·VS Code)에서도 정상적인 문서로 보인다.
2. **CommonMark/GFM과 충돌하지 않는다** — 본문에 새 문법을 발명하지
   않는다. 확장 정보는 본문 밖 메타데이터 계층에 둔다.
3. **구조와 스타일을 분리한다** — 헤딩은 글자 크기가 아니라 **노드
   레벨**이다. 시각 표현은 맵 정책(레벨 규칙)과 노드 오버라이드가
   담당한다.
4. **무손실 왕복** — 맵 → MD → 맵을 반복해도 구조·노트·스타일·레이아웃·
   설정·사진 위치가 보존된다.
5. **AI 입출력 친화** — LLM이 일반 Markdown만 생성해도 유효한 EMM
   문서다 (메타데이터는 선택 사항).
6. **도구 독립(tool-independent)** — 본 스펙과 적합성 코퍼스만으로
   제3자 파서/렌더러를 구현할 수 있어야 한다.

### 1.3 비목표 (v1.0)

- 좌표 픽셀 단위의 완전한 시각 재현 규격화 (레이아웃은 엔진 재계산)
- 협업 상태·권한·감사 로그의 파일 내 표현
- 본문 내 커스텀 지시자 문법(`:::`, `@key:` 등) — v1.0은 도입하지 않음

## 2. 설계 원칙 (Normative)

### 2.1 2계층 구조 — "본문은 순수 GFM, 충실도는 메타데이터로"

EMM 문서는 두 계층으로 구성된다.

```
┌─────────────────────────────────────────────┐
│  ① 본문 (Content Layer)                      │
│     100% CommonMark/GFM — 새 문법 없음        │
│     구조(헤딩·리스트)·노트(인용·표·펜스)·      │
│     링크·사진·인라인 강조                     │
├─────────────────────────────────────────────┤
│  ② 메타데이터 (Fidelity Layer) — 선택         │
│     파일 끝 HTML 주석 1줄:                    │
│     <!-- easymindmap:v1:BASE64(JSON) -->     │
│     스타일·레이아웃·레벨 정책·노트 원본·       │
│     사진 위치·맵 설정 — 무손실 복원용          │
└─────────────────────────────────────────────┘
```

이 구조의 결과:

- 메타데이터가 없어도 **유효한 EMM 문서**다 (구조·콘텐츠는 본문만으로
  복원). 손으로 쓴 일반 MD, AI가 생성한 MD가 즉시 EMM이 된다.
- 메타데이터가 있으면 **무손실 복원**이 보장된다.
- 일반 MD 뷰어에서 메타데이터는 HTML 주석이므로 **보이지 않는다** —
  본문 가독성을 해치지 않는다.

> 역사적 참고: 초기 설계의 "simple/full 이원 export"는 이 2계층 구조로
> 대체되어 폐기되었다. 단일 export가 곧 simple(본문)이면서
> full(메타데이터)이다.

### 2.2 헤딩 = 레벨 (스타일 아님)

- `#`~`######`은 노드의 **깊이**만 의미한다.
- 렌더러는 헤딩 레벨에 따른 글자 크기·굵기를 적용하지 **않는다**.
  글자 크기는 맵의 레벨 폰트 정책 → 서브트리/노드 오버라이드 순으로
  결정된다.
- 같은 레벨의 노드는 같은 기본 스타일을 가진다.

### 2.3 정책(policy)과 오버라이드(override)의 분리

- 레벨별 레이아웃·폰트·도형 등 "규칙"은 **맵 단위 정책**이며 메타데이터
  계층에 저장한다.
- 노드에는 정책과 **다른 값만**(오버라이드) 저장한다. 본문에 반복 기술을
  금지해 파일 비대화와 diff 오염을 막는다.

### 2.4 관용적 파싱 (Postel's law)

- 파서는 가능한 한 중단하지 않고 복구한다.
- 알 수 없는 메타데이터 필드는 무시한다 (전방 호환).
- 손상된 메타데이터는 버리고 본문만으로 구조를 복원한다.

## 3. 문서 구조 문법 (Content Layer — Normative)

이 장의 규칙은 레퍼런스 구현과 E2E 적합성 코퍼스로 검증된 **현행
규격**이다. 변환 표 단위의 세부 규칙과 예시는 `markdown-export.md`
§1~§2가 규범이다. 여기서는 규격의 뼈대를 요약한다.

### 3.1 노드 계층

| 본문 | 맵 |
|---|---|
| 파일의 첫 `#` | 루트(중심 주제) + 맵 제목 |
| 제목 이후 본문의 `#` | 2레벨 노드 (h1Mode: 본문 H1이 있으면 `#`=2레벨, `##`=3레벨…로 한 단계 내려 계층 보존) |
| `##`~`######` | 2~6레벨 노드 |
| 불릿 리스트(`-`/`*`/`+`, 들여쓰기 2칸=1단계) | 마지막 견출(또는 절)의 하위 노드 |
| 순번 리스트(`1.`/`1)`) | 하위 노드 — 번호를 텍스트에 유지 |
| 들여쓰기 없는 순번 항목 | **절 머리** — 다음 순번/견출 전까지의 문단·펜스가 그 절의 자식 |
| 7레벨 이상 | 리스트 들여쓰기로 표현 |

부모 결정: 현재 항목보다 얕은 가장 가까운 이전 노드가 부모다.
루트 없는 문서는 가상 루트를 생성한다. 빈 견출은 무시한다.

### 3.2 노드 콘텐츠 수집

- 견출/절 머리 뒤의 일반 문단(연속 줄)은 한 노드로 합쳐 그 하위가 된다.
- 문단 바로 뒤의 불릿은 그 문단 노드의 자식이 된다.
- 리스트 항목의 들여쓴 연속 줄은 항목 노드의 추가 줄로 병합된다.
- 첫 견출 전의 문단·표·인용문은 **루트의 노트**가 된다(머리말).

### 3.3 노트 (구조화 블록)

| 본문 | 맵 |
|---|---|
| `>` 인용문 | 현재 노드의 **문단 노트** |
| ``` 코드 펜스 ``` (언어 표기 유지) | 현재 노드의 **코드 노트** |
| 파이프 표 | 직전 노드의 **표 노트** (구분선 제거, `셀 | 셀` 줄 형식) |

노트는 노드 "본문"이 아니라 노드에 첨부된 구조화 블록이며, 렌더러는
인디케이터(T/C/⊞/✓)와 뷰어 팝업으로 표시한다.

### 3.4 링크·사진

| 본문 | 맵 |
|---|---|
| `[라벨](url)` (노드 텍스트 내) | 텍스트에는 라벨만, URL은 노드 링크(🔗)로 첨부 |
| `🔗 [라벨](url)` (export 형식) | 노드 링크 |
| `![대체](files/img-N.ext)` | 노드 사진 — export 시 내장 사진(base64)은 `files/`로 ZIP 패키징, 텍스트 중간 위치(afterLine)는 메타데이터가 보존 |
| 노트(인용문) 안 링크 | 원문 유지 — 뷰어가 앵커로 렌더링 |

### 3.5 인라인 강조

`**굵게**` `*기울임*` `~~취소선~~` `__밑줄__` `==하이라이트==`는 노드
텍스트에 원문 그대로 저장되고, 렌더러가 마커 문자를 숨기고 서식으로
표시한다. 백슬래시 이스케이프(`1\.`, `\-`)는 해제한다.

## 4. 메타데이터 계층 (Fidelity Layer — Normative)

### 4.1 형식

파일 마지막에 HTML 주석 한 줄:

```
<!-- easymindmap:v1:eyJ2ZXJzaW9uIjoxLCJtYXAiOnsi... -->
```

- 접두 `easymindmap:` + 스키마 버전 `v1` + `:` + **Base64(UTF-8 JSON)**.
- JSON 페이로드는 맵 전체 스냅숏이다: 맵 설정(레벨 폰트·레이아웃·도형
  정책, 노트 폰트, 테마), 노드별 오버라이드(스타일·도형·정렬·레이아웃·
  side·접힘·수동 크기), 노트 원본(리치 html 포함), 링크, 태그, 사진
  (내장 data URL·텍스트 중간 위치 afterLine), 첨부(≤2MB 인라인).
  상세 스키마는 `import-export/22-map-file-meta.md`.
- 사진·첨부가 큰 경우 `제목.md + files/` 구조의 ZIP으로 패키징된다.

### 4.2 처리 규칙

1. import 시 메타데이터가 있으면 **메타데이터가 우선**한다 — 본문은
   가독 표현, 메타데이터가 원본이다.
2. 메타데이터가 없거나 손상되면 본문만으로 §3 규칙에 따라 복원한다.
3. 알 수 없는 필드는 무시하고 보존 재기록하지 않아도 된다(v1).
4. `v1` 외 버전 접두는 향후 스키마 개정용으로 예약한다. 파서는 알 수
   없는 버전의 메타데이터를 무시하고 본문 복원으로 폴백해야 한다.

## 5. 적합성 (Conformance)

구현체는 두 수준 중 하나를 선언할 수 있다.

### 5.1 EMM-Basic (구조 적합)

- §3 본문 문법으로 트리·노트·링크를 읽고/쓸 수 있다.
- 메타데이터 계층은 **무시해도 된다** (단, 파일을 재기록할 때 기존
  메타데이터 주석을 삭제하지 않고 유지하는 것을 권장).
- 일반 Markdown 도구는 사실상 EMM-Basic 뷰어다.

### 5.2 EMM-Full (무손실 적합)

- EMM-Basic + §4 메타데이터 계층의 기록·복원.
- 왕복 보장: `맵 → MD → 맵` 후 구조·노트·스타일·레이아웃·설정·사진이
  동일해야 한다.

### 5.3 적합성 검증 코퍼스

적합성 코퍼스는 **`packages/emm-parser/conformance/`**의 정식 자산이다:
실문서 7종(전자영수증 보고서 176노드 · ChatGPT 대화 299노드 · 견출 없는
순번 절 96노드 · README 배지 링크 · 회의록 h1Mode · 깊은 견출 · 머리말만
문서) + AI 프롬프트 기대 출력 4종 = 11케이스. 케이스마다 ▸파싱 결과
스냅숏 ▸메타데이터 무손실 왕복 ▸본문 왕복 노드 수를 자동 검증한다
(`npm test`). **제3자 구현체는 이 코퍼스를 통과하면 EMM 호환을 선언할
수 있으며, 변환 규칙 변경은 코퍼스 전체 통과 + 문서 갱신을 조건으로
한다.**

### 5.4 오류 처리 원칙

| 상황 | 처리 |
|---|---|
| 루트 없음 | 가상 루트 생성 |
| 헤딩 레벨 건너뜀 (`#` → `###`) | 중간 레벨을 만들지 않고 바로 하위로 연결 |
| 빈 헤딩 | 무시 |
| H6 초과 깊이 | 리스트 들여쓰기로 표현 |
| 미종료 코드 펜스 | 파일 끝까지를 코드로 수용 |
| 메타데이터 손상 | 폐기 후 본문 복원 |

## 6. 버전 관리

- 본 스펙: `EMM v1.0` — 의미적 변경 시 minor/major 증가.
- 메타데이터 스키마: `easymindmap:v1` 접두 — 하위 호환이 깨지는 변경만
  `v2`로 올린다. v1 파서는 미지 버전을 본문 폴백으로 처리해야 한다.
- 하위 호환 원칙: 이미 내보낸 파일은 이후 버전의 EasyMindMap에서 항상
  열려야 한다.

## 7. 확장 로드맵 (Informative)

v1.0 이후 확장 후보 — 모두 **additive**(기존 문서 유효성 유지)여야 한다.

| 확장 | 방향 |
|---|---|
| 크로스 엣지 (트리 외 연결선) | 메타데이터 계층에 `edges[]` — 본문 문법 불변 |

## 8. 제3자 구현 가이드 (Informative)

1. **최소 구현(EMM-Basic)**: CommonMark 파서로 헤딩 트리를 만들고
   §3.1~3.2의 부모 결정·콘텐츠 수집 규칙을 얹으면 된다. 인용/표/펜스를
   노드의 노트로 분류하면 끝.
2. **무손실 구현(EMM-Full)**: 파일 끝 `<!-- easymindmap:v1:...` 주석을
   찾고 Base64→JSON 파싱. 스키마는 22-map-file-meta.md.
3. 렌더링 시 헤딩 크기를 적용하지 말 것 (§2.2).
4. 재기록 시 자신이 이해하지 못한 메타데이터 주석을 버리지 말 것.
