과거에는 모든 데이터를 Redux 같은 하나의 거대한 저장소에 담았지만, 현대적인 방식은 \*\*"데이터의 성격"\*\*에 따라 도구를 나누어 사용하는 것입니다.

1\. TanStack Query (서버 상태 관리)
-----------------------------

\*\*서버에서 가져온 데이터(Server State)\*\*를 관리하는 데 최적화된 라이브러리입니다.

*   **주요 역할:** API 호출, 캐싱, 데이터 동기화, 업데이트.
*   **핵심 기능:**
    *   **캐싱(Caching):** 동일한 API 요청 시 서버에 다시 묻지 않고 기존 데이터를 즉시 반환합니다.
    *   **자동 리프레시:** 사용자가 브라우저 탭을 다시 눌렀을 때(focus), 또는 일정 시간이 지났을 때 자동으로 최신 데이터를 가져옵니다.
    *   **상태 추적:** `isLoading`, `isError`, `data` 등의 상태를 자동으로 제공하여 로딩 스피너나 에러 처리를 쉽게 구현하게 해줍니다.
    *   **낙관적 업데이트(Optimistic Updates):** 서버 응답이 오기 전 화면을 먼저 업데이트하여 사용자 경험을 높입니다.

* * *

2\. Zustand (클라이언트 상태 관리)
-------------------------

\*\*브라우저 내부에서만 유지되는 데이터(Client State)\*\*를 관리하는 가볍고 빠른 라이브러리입니다.

*   **주요 역할:** UI 테마, 로그인한 유저 정보, 모달 열림/닫힘, 장바구니 등.
*   **핵심 기능:**
    *   **단순함:** Redux처럼 복잡한 Boilerplate(Action, Reducer 등)가 거의 없습니다. 훅(Hook) 기반으로 직관적입니다.
    *   **성능:** 상태의 특정 부분만 구독(Selector)할 수 있어, 불필요한 리렌더링을 방지합니다.
    *   **유연성:** 특정 라이브러리에 종속되지 않아 React 없이도 사용 가능하며, Context API보다 설정이 간단합니다.

결론부터 말씀드리면, **두 도구는 경쟁 관계가 아니라 상호보완 관계**입니다. 하나를 고르는 것이 아니라 **둘을 함께 사용하는 것이 베스트 프랙티스**입니다.

### **데이터 성격에 따른 분류**

| 구분  | 서버 데이터 (Server State) | 클라이언트 데이터 (Client State) |
| --- | --- | --- |
| **특징** | DB에 저장됨, 비동기 요청 필요, 타인과 공유됨 | 브라우저 메모리에 존재, 즉각적, 로컬 설정 |
| **예시** | 사용자 목록, 게시글 내용, 주문 내역 | 다크모드 여부, 사이드바 개폐, 입력 폼 중간 저장 |
| **추천 도구** | **TanStack Query** | **Zustand** |

### **추천하는 아키텍처**

1.  **대부분의 데이터는 TanStack Query로 관리하세요.** \* 우리가 관리하는 상태의 80% 이상은 사실 서버에서 가져온 데이터입니다. 이를 Query가 담당하면 관리 포인트가 획기적으로 줄어듭니다.
2.  **전역적으로 공유가 필요한 UI 상태만 Zustand에 담으세요.**
    *   서버 데이터가 아닌 "순수 UI 상태"만 Zustand로 관리하면 코드가 매우 깔끔해집니다.

### 1\. 역할의 명확한 분리 (Client vs Server)

이 프로젝트는 데이터의 성격에 따라 관리 도구를 엄격히 구분하고 있습니다.

*   **TanStack Query (서버 상태 관리):**
    *   **대상:** 마인드맵 데이터(노드 정보, 트리 구조), 사용자 프로필 등 서버(DB)에서 가져오는 데이터.
    *   **이유:** 비동기 데이터의 로딩(Loading), 에러(Error) 처리, 그리고 \*\*캐싱(Caching)\*\*을 효율적으로 하기 위함입니다. 특히 마인드맵처럼 데이터 변경이 잦은 경우, 서버와의 동기화 로직을 직접 구현하는 대신 라이브러리에 위임하여 안정성을 높였습니다.
*   **Zustand (클라이언트 상태 관리):**
    *   **대상:** 현재 선택된 노드 ID, 사이드바 개폐 여부, 편집 모드 상태(Edit Mode), 줌(Zoom) 레벨 등 UI 전용 상태.
    *   **이유:** 프로젝트가 복잡해짐에 따라 Props Drilling(데이터를 하위로 계속 전달하는 현상)을 방지하고, Context API보다 가볍고 단순한 코드로 전역 UI 상태를 관리하기 위해 채택되었습니다.

### 2\. 구현 상의 특징

`05-implementation` 문서 및 관련 코드를 통해 확인된 주요 구현 패턴입니다.

*   **커스텀 훅(Custom Hooks) 패턴:** API 호출 로직을 `useGetMindmap`, `useUpdateNode`와 같은 TanStack Query 기반 커스텀 훅으로 추상화하여 컴포넌트 내부 코드를 깔끔하게 유지합니다.
*   **Selective Update:** Zustand의 `selector` 기능을 활용하여, 특정 상태가 변경될 때 해당 상태를 사용하는 컴포넌트만 리렌더링되도록 최적화되어 있습니다.
*   **Persistence (선택적):** UI 설정값 중 일부(예: 다크모드)는 Zustand의 `persist` 미들웨어를 사용하여 브라우저 재시작 시에도 유지되도록 구성되어 있을 가능성이 높습니다.

`easymindmap`은 현대적인 React 개발의 표준이라고 할 수 있는 **"Server State는 Query로, Client State는 Zustand로"** 나누어 관리하는 방식을 충실히 따르고 있습니다. 이 방식은 코드의 가독성을 높일 뿐만 아니라, 마인드맵과 같이 복잡한 UI 인터랙션이 필요한 서비스에서 성능을 최적화하는 데 매우 유리한 선택입니다.

1\. TanStack Query (서버 상태 관리자)
------------------------------

**역할: "서버(DB)라는 멀리 있는 창고에서 물건을 가져오고 관리하는 배달원"**

여러분이 쇼핑몰 앱을 만든다고 가정해 봅시다. 상품 목록은 내 컴퓨터가 아니라 서버(DB)에 저장되어 있습니다.

*   **기능 1: 배달 (Fetching)** 서버에 "상품 목록 좀 보내줘!"라고 요청해서 받아옵니다.
*   **기능 2: 중간 보관 (Caching)** 한 번 받아온 데이터는 잠깐 자기 가방(캐시)에 넣어둡니다. 사용자가 페이지를 나갔다 다시 들어와도, 서버에 다시 묻지 않고 가방에서 바로 꺼내 보여줍니다. (매우 빠름!)
*   **기능 3: 신선도 체크 (Invalidation)** 가방에 있는 데이터가 너무 오래되었으면 알아서 버리고 서버에서 새 데이터를 가져옵니다. "이 데이터 아직 신선한가?"를 대신 고민해 주는 거죠.
*   **기능 4: 로딩/에러 처리** "지금 데이터를 가져오는 중이야(Loading)", "가져오다 실패했어(Error)"라는 상태를 자동으로 알려주어 화면에 빙글빙글 도는 스피너를 띄우기 쉽게 해줍니다.

* * *

2\. Zustand (클라이언트 상태 관리자)
--------------------------

**역할: "내 브라우저(내 방) 안에서만 쓰는 물건들을 정리하는 수납장"**

서버와 상관없이, 지금 당장 내 화면에서 벌어지는 일들을 기억하는 곳입니다.

*   **기능 1: 전역 공유 (Global Store)** 만약 '다크모드 설정' 정보를 아주 깊숙한 곳에 있는 컴포넌트에 전달해야 한다면, 일일이 부모-자식 간에 전달(Props Drilling)할 필요 없이 **Zustand라는 공용 수납장**에 넣어두고 필요한 사람만 꺼내 쓰면 됩니다.
*   **기능 2: 상태 기억 (Client State)** "사이드바가 열려 있는가?", "사용자가 입력 폼에 무엇을 타이핑했는가?" 같은 휘발성 정보를 저장합니다.
*   **기능 3: 매우 가벼움** 설정이 아주 단순해서 초보자도 "수납장 이름 정하기 -\> 물건 넣기 -\> 물건 꺼내기" 3단계면 바로 쓸 수 있습니다.

4\. 실제 개발에서의 흐름 (EasyMindMap 예시)
--------------------------------

여러분이 참고하신 마인드맵 프로젝트를 예로 들어볼게요.

1.  사용자가 접속하면 **TanStack Query**가 서버에서 '마인드맵 데이터'를 배달해 옵니다. (로딩 중엔 빙글빙글 스피너를 보여줌)
2.  사용자가 특정 노드(원)를 클릭합니다. 이때 "지금 3번 노드가 선택됨"이라는 정보는 **Zustand**라는 수납장에 저장합니다. (다른 화면 요소들이 "아, 3번이 선택됐구나!"라고 알 수 있게 됨)
3.  사용자가 노드 이름을 수정하고 저장 버튼을 누르면, **TanStack Query**가 다시 서버로 달려가서 "이거 수정됐어!"라고 알려주고 데이터를 최신으로 업데이트합니다.

### 요약하자면:

*   **TanStack Query**는 \*\*"서버 데이터"\*\*를 관리하는 똑똑한 배달원입니다.
*   **Zustand**는 \*\*"UI 상태"\*\*를 관리하는 간편한 수납장입니다.

이 두 가지만 잘 다뤄도 현대적인 웹 개발의 상태 관리 문제는 90% 이상 해결됩니다. 처음에는 어렵게 느껴질 수 있지만, "데이터가 어디서 오는가?"만 생각해보면 어떤 패키지를 쓸지 쉽게 결정할 수 있을 거예요.

혹시 이 중에서 좀 더 구체적인 사용법(코드 예시 등)이 궁금한 부분이 있으신가요?

## Prompt:
두개의 솔루션은 어떤 환경에서 실행되는건지 자세히 설명해줘

## Response:
Zustand와 TanStack Query가 \*\*"어디서, 어떻게 실행되는지"\*\*를 이해하면 개발할 때 전체적인 그림을 그리는 데 큰 도움이 됩니다. 두 솔루션은 모두 \*\*브라우저(Client-Side)\*\*에서 실행되지만, 데이터를 관리하는 \*\*생명 주기(Lifecycle)\*\*와 **목적지**가 다릅니다.

1\. 실행 환경: 브라우저의 메모리 (RAM)
--------------------------

두 라이브러리 모두 사용자의 브라우저(Chrome, Safari 등)가 실행되는 동안 **컴퓨터의 메모리(RAM)** 공간을 빌려서 데이터를 저장합니다.

*   **Zustand:** 브라우저 메모리 안에 작은 \*\*"전역 변수 보관함"\*\*을 만듭니다.
*   **TanStack Query:** 브라우저 메모리 안에 \*\*"임시 저장소(캐시)"\*\*를 만듭니다.

> \[!IMPORTANT\] **공통점:** 브라우저를 '새로고침'하면 메모리가 초기화되기 때문에 기본적으로는 데이터가 사라집니다. (물론 이를 방지하는 설정도 가능합니다.)

* * *

2\. TanStack Query의 실행 메커니즘 (브라우저 ↔ 서버)
---------------------------------------

TanStack Query는 브라우저 안에서 실행되지만, 항상 \*\*"외부 서버"\*\*를 바라보고 있습니다.

*   **실행 위치:** 브라우저 내부.
*   **작동 방식:**
    1.  브라우저가 켜지면 TanStack Query가 서버에 데이터를 요청합니다.
    2.  데이터가 도착하면 브라우저 메모리의 \*\*'캐시 저장소'\*\*에 저장합니다.
    3.  컴포넌트가 데이터를 필요로 하면 서버에 다시 가지 않고 이 \*\*'캐시 저장소'\*\*에서 데이터를 꺼내줍니다.
*   **특징:** 네트워크 상태(인터넷 연결 등)에 민감하게 반응하며 실행됩니다.

* * *

3\. Zustand의 실행 메커니즘 (브라우저 내부 전용)
---------------------------------

Zustand는 서버를 신경 쓰지 않습니다. 오직 \*\*"현재 켜져 있는 웹 페이지 내부"\*\*에서만 작동합니다.

*   **실행 위치:** 브라우저 내부 (컴포넌트 외부의 독립된 공간).
*   **작동 방식:**
    1.  자바스크립트가 실행될 때 메모리에 특정 공간을 확보합니다.
    2.  A라는 컴포넌트가 "다크모드 켜줘!"라고 Zustand에 신호를 보냅니다.
    3.  Zustand는 메모리 값을 바꾸고, 이 값을 지켜보고 있던 B, C 컴포넌트에게 "값이 바뀌었으니 화면을 다시 그려!"라고 명령합니다.
*   **특징:** 서버와 통신하지 않으므로 즉각적(0.0001초)으로 반응합니다.

🛠 상태 관리 설계 (State Management Design)
=====================================

이 문서는 `EasyMindMap` 프로젝트에서 **TanStack Query**와 **Zustand**를 상태 관리 도구로 선택한 이유와 각 도구의 역할 분담 체계를 설명합니다.

1\. 상태 관리 전략 개요
---------------

현대적인 웹 애플리케이션의 데이터는 그 성격에 따라 두 가지로 분류됩니다. 본 프로젝트는 이 두 데이터를 혼합하지 않고 각각 최적화된 도구를 사용하여 관리합니다.

1.  **Server State (서버 상태):** 원격 데이터베이스에 저장되어 있으며, API를 통해 가져와야 하는 데이터 (예: 마인드맵 구조, 노드 정보, 유저 프로필).
2.  **Client State (클라이언트 상태):** 브라우저 세션 내에서만 유효하며, UI의 인터랙션을 제어하는 데이터 (예: 선택된 노드 ID, 사이드바 개폐 여부, 줌 레벨).

* * *

2\. TanStack Query (서버 상태 관리)
-----------------------------

### 도입 이유

마인드맵 데이터는 사용자의 편집에 따라 빈번하게 서버와 동기화되어야 합니다. 직접 비동기 로직을 구현할 경우 발생할 수 있는 캐싱 오류와 복잡한 상태 업데이트를 해결하기 위해 도입했습니다.

### 주요 역할

*   **캐싱 및 동기화:** 동일한 마인드맵 조회 시 네트워크 요청을 최소화하고, 수정 시 캐시를 무효화하여 최신 상태를 유지합니다.
*   **비동기 상태 표준화:** `isLoading`, `isError` 등의 상태를 제공하여 일관된 로딩/에러 UI를 구현합니다.
*   **데이터 낙관적 업데이트(Optimistic Updates):** 노드 이동이나 수정 시, 서버 응답 전에 화면을 먼저 업데이트하여 사용자 경험(UX)을 극대화합니다.

* * *

3\. Zustand (클라이언트 상태 관리)
-------------------------

### 도입 이유

Redux와 같은 복잡한 Boilerplate 코드 없이, 가볍고 직관적으로 전역 UI 상태를 관리하기 위해 도입했습니다. Context API의 리렌더링 성능 이슈를 해결하면서도 학습 곡선이 낮다는 장점이 있습니다.

### 주요 역할

*   **UI 인터랙션 관리:** 현재 어떤 노드가 강조(Focus)되어 있는지, 메뉴 바가 열려 있는지 등의 순수 UI 상태를 저장합니다.
*   **컴포넌트 간 통신:** 깊은 구조의 컴포넌트 트리에서 Props Drilling 없이 상태를 공유합니다.
*   **성능 최적화:** Selector 패턴을 사용하여 필요한 상태가 변경될 때만 해당 컴포넌트가 리렌더링되도록 제어합니다.

==============================================

이 문서는 `EasyMindMap` 프로젝트가 왜 **TanStack Query**와 **Zustand**를 선택했는지, 그리고 초보 개발자도 이해할 수 있는 수준의 동작 원리와 실행 환경에 대해 상세히 기술합니다.

1\. "상태(State)"란 무엇인가?
----------------------

웹 개발에서 \*\*상태(State)\*\*는 \*\*"시간이 흐름에 따라 변할 수 있는 모든 데이터"\*\*를 의미합니다.

*   **예시:** 로그인한 사용자의 이름, 현재 열려있는 마인드맵 노드, 화면의 다크모드 여부 등.
*   **비유:** 요리(웹 서비스)를 할 때 필요한 \*\*식재료(데이터)\*\*와 같습니다. 식재료가 신선한지, 어디에 보관되어 있는지 관리하는 것이 바로 **상태 관리**입니다.

* * *

2\. 도구별 역할 분담: "배달원"과 "수납장"
---------------------------

본 프로젝트는 데이터의 출처에 따라 역할을 엄격히 분리합니다.

### 🚚 TanStack Query (서버 상태 관리자)

**"서버라는 먼 창고에서 데이터를 가져오는 전문 배달원"**

*   **주요 기능:**
    *   **Fetching:** 서버 API를 호출하여 데이터를 가져옵니다.
    *   **Caching:** 한 번 가져온 데이터는 메모리에 임시 저장하여 중복 요청을 방지합니다.
    *   **Synchronization:** 서버의 데이터가 변경되면 화면을 자동으로 최신화합니다.
    *   **Status Tracking:** 로딩 중(`isLoading`), 에러 발생(`isError`) 상태를 자동으로 관리합니다.
*   **대상 데이터:** 마인드맵 노드 구조, 사용자 프로필, 공유 문서 목록 등 (DB 저장 데이터).

### 🗄️ Zustand (클라이언트 상태 관리자)

**"내 방(브라우저) 안의 물건들을 정리하는 가벼운 수납장"**

*   **주요 기능:**
    *   **Global Store:** 컴포넌트 간에 데이터를 직접 주고받지 않고, 공용 수납장에서 누구나 꺼내 쓸 수 있게 합니다.
    *   **Instant Update:** 서버를 거치지 않으므로 즉각적인 UI 반응을 제공합니다.
    *   **Simplicity:** 코드가 매우 직관적이며 설정이 간단합니다.
*   **대상 데이터:** 사이드바 개폐 여부, 현재 선택된 노드 ID, 줌 레벨, 에디터 설정 등 (UI 전용 데이터).

* * *

3\. 실행 환경 및 기술 스택
-----------------

### 🌐 실행 환경: 브라우저 메모리 (RAM)

두 솔루션 모두 **사용자의 브라우저(Client-Side)** 내 메모리 공간에서 실행됩니다.

*   **공통점:** 새로고침 시 기본적으로 초기화되지만, 브라우저의 전용 공간(RAM)을 효율적으로 점유하여 앱의 속도를 높입니다.
*   **차이점:** TanStack Query는 항상 **네트워크(인터넷)** 상태를 주시하며 서버와 소통하는 반면, Zustand는 오직 **사용자의 클릭/입력**에만 반응합니다.

### ⌨️ 사용 언어: JavaScript & TypeScript

본 프로젝트는 **JavaScript**를 기반으로 하되, 안정성을 위해 **TypeScript** 환경에서 실행됩니다.

*   **JavaScript:** 브라우저의 기본 언어로, 모든 로직의 뼈대를 이룹니다.
*   **TypeScript:** 데이터의 '타입(형태)'을 정의합니다. 예를 들어, 마인드맵 노드가 `string`인지 `number`인지 미리 정의하여 개발자의 실수를 컴파일 단계에서 방지합니다.
